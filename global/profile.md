# 用户画像

> **定位**：Agent 在涉及写作、总结、分享稿、知识库规则调整、判断取舍时读取本文件。
> **原则**：参考我的偏好，不伪装成我本人。产出应该是"我审过的内容"，不是"我的口吻"。
> **维护**：由用户本人维护和更新。Agent 不能自行修改。

## 我是谁

- （你的角色：开发者？项目经理？独立创作者？）
- 当前主力项目：（列 2-3 个）
- 技术栈偏好：（后端语言、前端框架、工具链等）

## 我的表达习惯

- **我习惯的语言**：（中文/英文？口语化还是正式？）
- **我会写的东西**：（技术方案？产品说明？决策记录？——实用导向的）
- **我不会写的东西**：（空洞的"优化建议"？热情营销语气？过度修饰？）
- **我接受的风格**：（直接、简洁、有依据。说清楚"是什么、为什么、怎么做"）

## 我的决策偏好

- **先小后大**：先验证一个点，确认能跑通，再铺开。
- **先稳后新**：现有的能用就别急着重构。新东西先实验验证。
- **讨厌的事**：
  - 不经明确指示擅自修改内容、扩大范围、改动关键文件。
  - 不经明确要求自行提交代码、推送、创建 PR。
  - 虚报工作完成度——没验证就说完成、没跑通就说通过。
  - 复杂任务闷头一次性做完，中间不对齐，最后给不出明确方案。
  - 跳过验证步骤，把临时方案包装成最终方案。
- **重视的事**：规则的一致性、决策的追溯（为什么要这样做）、受保护文件的边界。

## Agent 协作边界

以下事情 Agent 可以做：
- 执行具体任务（写代码、查问题、跑验证）
- 提供方案选项和各自优缺点
- 按已有规则产出内容
- 提醒我遗漏的事项

以下事情必须我本人拍板：
- 最终方案选择
- 知识库规则修改
- 受保护文件修改
- 外部工具纳入默认流程
- 跨项目决策

## 输出禁忌

Agent 产出的内容不能出现：

- **假冒口吻**：不要用"我认为""我觉得"冒充我的判断。Agent 的判断用"建议""分析显示"来表述。
- **过度承诺**："完美""万无一失""绝对"——没有这种事。
- **空洞总结**：每一条结论要接依据，不要写"整体来说做得不错"这种没法验证的话。
- **隐形越界**：不能在产出内容里悄悄加入"因此建议改为"之类的越权判断。
